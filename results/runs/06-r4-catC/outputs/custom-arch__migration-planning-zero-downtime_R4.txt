Key findings: The service currently runs on a single MySQL 8 primary with no replicas, handles about 2,000 writes/minute and 50,000 reads/minute at peak, and stores about 400 GB of data. A zero-downtime migration to PostgreSQL 16 requires online schema conversion, initial backfill while writes continue, continuous change replication, a dual-write or CDC validation phase, strong consistency checks before cutover, feature-flagged traffic switching, and a tested rollback path.

Step-by-step migration plan:

1. Discovery and planning
- Inventory all MySQL objects: tables, indexes, constraints, foreign keys, triggers, views, stored procedures, events, users, grants, and application queries.
- Identify high-risk tables and critical business entities.
- Capture current performance baselines: write rate, read rate, query latency, slow queries, lock behavior, storage growth, and peak traffic periods.
- Define success criteria: no customer-facing downtime, PostgreSQL replication lag near zero before cutover, no unresolved data mismatches, acceptable application error rate, and PostgreSQL able to sustain production load.
- Establish migration feature flags: read-from-MySQL, read-from-PostgreSQL, MySQL-only writes, dual-write, PostgreSQL-only writes, and rollback-to-MySQL.

2. Build the PostgreSQL 16 target environment
- Provision PostgreSQL 16 with enough CPU, memory, IOPS, storage, and network capacity for at least peak production load plus migration overhead.
- Configure backups, point-in-time recovery, monitoring, alerting, connection pooling, and operational runbooks before loading production data.
- Monitor PostgreSQL CPU, memory, disk, connections, locks, replication/apply lag, query latency, deadlocks, autovacuum, and write throughput.

3. Translate and validate the schema
- Convert MySQL DDL to PostgreSQL-compatible DDL using tools such as pgloader, AWS Schema Conversion Tool, or custom scripts.
- Address key MySQL-to-PostgreSQL differences:
  - MySQL ENUM to PostgreSQL ENUM or lookup/reference tables.
  - TINYINT(1) used as boolean to PostgreSQL BOOLEAN.
  - AUTO_INCREMENT to GENERATED BY DEFAULT/ALWAYS AS IDENTITY.
  - MySQL case-insensitive behavior and identifier handling to PostgreSQL case-sensitive identifiers and appropriate collations.
  - MySQL-specific functions, indexes, collations, DDL syntax, JSON behavior, date/time behavior, and upsert syntax.
- Recreate primary keys, foreign keys, unique constraints, indexes, and check constraints appropriately in PostgreSQL.
- Validate the converted schema in staging before touching production data.

4. Prepare the application
- Update application code to be compatible with both databases during the migration.
- Abstract database-specific SQL differences, including pagination, upserts, JSON operators, boolean handling, timestamp precision, transaction behavior, and locking semantics.
- Make writes idempotent where possible.
- Add durable write IDs, versions, timestamps, or transaction identifiers to support ordering, deduplication, and divergence detection.
- Add instrumentation to compare MySQL and PostgreSQL behavior during shadow reads and dual-write.

5. Perform initial backfill without stopping writes
- Keep MySQL as the system of record.
- Start an online snapshot/backfill from MySQL to PostgreSQL while MySQL continues to serve production reads and writes.
- Use an online migration mechanism such as AWS Database Migration Service, Debezium CDC with snapshotting, pgloader plus CDC, or another snapshot-plus-change-data-capture pipeline.
- Copy the 400 GB dataset in chunks to avoid long locks and minimize impact on the single MySQL primary.
- Throttle the backfill if MySQL latency, lock contention, or replication/apply lag becomes risky.
- Record the MySQL binlog position or CDC offset at the start of the snapshot so all writes that occur during the backfill can be replayed into PostgreSQL.

6. Enable continuous replication and catch-up
- After the initial snapshot, continuously apply MySQL binlog changes into PostgreSQL using CDC.
- Keep MySQL authoritative during this phase.
- Monitor replication lag, failed events, transformation errors, duplicate-key errors, data type conversion issues, and PostgreSQL write capacity.
- Do not proceed until PostgreSQL is caught up, lag is stable and low, and all failed events have been resolved.

7. Validate data before cutover
- Compare row counts for all tables.
- Compare primary key ranges and distributions.
- Run chunked checksums on large tables and full checksums on smaller or critical tables.
- Validate business invariants such as account balances, order totals, user states, referential relationships, and other domain-specific consistency rules.
- Compare sampled records between MySQL and PostgreSQL.
- Run application-level query comparisons in staging or shadow mode.
- Any discrepancy must be explained, repaired, and revalidated before cutover.

8. Run shadow reads and performance testing
- Continue serving customer reads from MySQL.
- In parallel, issue shadow reads to PostgreSQL for selected requests and compare results.
- Account for expected differences such as timestamp precision, ordering without explicit ORDER BY, collation behavior, or case sensitivity.
- Tune PostgreSQL indexes, query plans, autovacuum, memory settings, connection pooling, and slow queries.
- Gradually route low-risk read traffic to PostgreSQL only after validation and performance are acceptable.

9. Start the dual-write phase
- Deploy application code that can write to both MySQL and PostgreSQL behind a feature flag.
- Initially keep MySQL as the source of truth.
- Enable dual-write for a small percentage of traffic, then gradually increase to 100% if metrics remain healthy.
- Recommended write pattern during the initial dual-write phase:
  - Write to MySQL first.
  - Write to PostgreSQL second.
  - Use idempotency keys or write versions to avoid duplicates.
  - Retry transient PostgreSQL failures.
  - Send persistent failures to a dead-letter queue for repair.
- Ensure ordering using transaction IDs, versions, timestamps, or CDC offsets so stale writes cannot overwrite newer values.
- If CDC is still active, prevent double-application of the same change through idempotency, deduplication, or a clearly defined handoff from CDC to application dual-write.
- Continue row-count checks, checksums, recent-write comparisons, business reconciliation, and error-queue monitoring throughout this phase.

10. Pre-cutover checklist
Before cutover, confirm all of the following:
- PostgreSQL contains a complete and validated copy of MySQL data.
- CDC or dual-write lag is effectively zero.
- No unresolved failed replication or dual-write events remain.
- Data validation and business reconciliation pass.
- PostgreSQL can handle production read and write load.
- Application behavior is stable in dual-write and/or shadow-read mode.
- Dashboards and alerts are active.
- Backups and point-in-time recovery are working for both systems.
- Rollback has been tested.
- Stakeholders have approved the cutover window.

11. Zero-downtime cutover procedure
- Keep the application online and perform the cutover using feature flags or routing changes.
- Announce a controlled migration window, but do not take the service down.
- Optionally gate or briefly pause new writes for only the shortest possible period, ideally seconds, to drain in-flight writes and eliminate final lag.
- Confirm final CDC/dual-write queues are fully drained.
- Run final validation: zero lag, no failed writes, critical row counts/checksums pass.
- Switch the write source of truth from MySQL to PostgreSQL using the feature flag.
- Route reads to PostgreSQL after or alongside the write cutover, depending on confidence from shadow-read testing.
- Keep MySQL online as a rollback target, preferably read-only or protected from accidental writes.
- Closely monitor application error rate, latency, PostgreSQL resource usage, lock contention, connection saturation, failed writes, and consistency checks.

12. Rollback criteria
Rollback if any of the following occur:
- Significant unexplained data inconsistency.
- PostgreSQL cannot sustain production traffic.
- Application error rate or latency exceeds agreed thresholds.
- Critical workflows fail.
- Writes fail after cutover.
- Replication, dual-write, or repair queues diverge or cannot catch up.
- Operational issues threaten customer impact.

13. Rollback procedure
- Before cutover: keep MySQL as source of truth, disable PostgreSQL reads/writes, fix the issue, rebuild or resynchronize PostgreSQL if needed, and retry only after validation passes.
- During dual-write: disable the dual-write flag, continue serving from MySQL, inspect or discard PostgreSQL changes, repair divergence, and revalidate before retrying.
- Immediately after cutover: if MySQL is still sufficiently current, switch reads and writes back to MySQL via feature flags, stop PostgreSQL writes, and investigate. If PostgreSQL accepted writes that MySQL did not, either replay those PostgreSQL writes back into MySQL or restore MySQL from backup and apply captured post-cutover changes.
- Do not decommission MySQL until PostgreSQL has been stable for an agreed period and rollback is no longer required.

14. Post-cutover stabilization
- Continue enhanced monitoring for at least several days to weeks.
- Continue consistency checks between MySQL and PostgreSQL while MySQL is retained.
- Keep MySQL available as a rollback target temporarily.
- Remove dual-write and compatibility code only after PostgreSQL stability is proven.
- Update runbooks, dashboards, backup procedures, incident response documentation, and developer guidance.
- Decommission MySQL only after business and engineering approval.

Estimated timeline:
- Discovery, schema conversion, and application preparation: about 2 weeks.
- Initial backfill and CDC setup: about 1 week, depending on throughput and production impact.
- Dual-write, shadow reads, validation, and tuning: about 4 weeks.
- Cutover preparation and approvals: several days.
- Cutover execution: 1 day.
- Post-cutover monitoring before decommissioning MySQL: 1 to 2 weeks or longer depending on risk tolerance.

Conclusion: The safest zero-downtime approach is to keep MySQL authoritative while building PostgreSQL 16 in parallel, translate and test the schema, perform an online backfill plus CDC, validate data with row counts/checksums/business checks, run shadow reads and dual-write behind feature flags, cut over only when lag is zero and validation passes, and retain MySQL temporarily with a tested rollback procedure.
