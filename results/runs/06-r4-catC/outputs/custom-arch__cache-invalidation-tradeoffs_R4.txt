Below is a self-contained comparison of four Redis cache invalidation strategies for a read-heavy HTTP API backed by a SQL database.

Key framing: cache correctness depends on what the cache is allowed to serve after a database write. In most real systems, Redis and SQL are not updated in a single true distributed transaction, so “atomic” cache correctness is harder than it sounds. Staleness bounds are usually expressed in terms of TTL, propagation lag, or the availability of an authoritative version check.

## 1. TTL-only / lazy expiration, no explicit invalidation

### Correctness properties
- Readers can see stale values after a database write.
- Normal worst-case staleness is approximately the configured TTL, assuming the cached item already existed before the write.
- If TTL is 5 minutes, a reader may see the old value for up to about 5 minutes.
- If TTLs are accidentally too long, missing, or refreshed incorrectly, stale data may last much longer.
- There is no immediate relationship between a DB write and cache invalidation.

### Operational complexity
- Lowest complexity.
- Requires only Redis plus application logic for cache-aside reads:
  1. read from Redis;
  2. on miss, read from SQL;
  3. populate Redis with a TTL.
- Main tuning burden is choosing TTLs that balance freshness, load, and hit rate.
- Common extra concerns: cache stampedes, hot keys, and TTL jitter to avoid synchronized expiration.

### Dominant failure modes
- Cache outage: application falls back to SQL if designed that way; DB load may spike and cause cascading failure.
- DB outage: cache hits may still work until TTL expiry, but misses and refreshes fail. This can provide partial degraded service.
- Replication lag: if cache fill reads from a lagging replica, Redis may cache an already-stale value for the full TTL.
- Bug in write path: since writes do not touch the cache, write-path cache bugs are less likely. But application bugs that set wrong TTLs, wrong keys, or cache incorrect values can persist until expiration.

### Right choice when
- Data can tolerate bounded staleness.
- The product is early-stage or operational simplicity matters more than freshness.
- Reads are heavy, writes are relatively infrequent, and stale reads are acceptable.
- Examples: public profiles, product catalogs, feature flags with conservative TTLs, non-critical metadata, dashboards where slight delay is acceptable.

### Wrong choice when
- Users must read their own writes immediately.
- Stale values can cause financial, security, authorization, inventory, or compliance problems.
- The TTL required for correctness is so short that the cache provides little benefit.

## 2. Write-through: DB write plus cache update from the application

In this strategy, every application write updates SQL and also updates or invalidates Redis from the write path.

### Correctness properties
- In the ideal case, readers see fresh cache values immediately after a successful write.
- However, SQL and Redis generally do not share a true atomic transaction. Unless using careful transaction/outbox/compensation logic, there are failure windows.
- Common ordering options:
  - DB first, then cache update: if DB commit succeeds but cache update fails, Redis may serve stale data until TTL, retry, manual repair, or another write.
  - Cache first, then DB commit: if cache update succeeds but DB write fails, Redis may temporarily contain data that never committed.
  - Delete/invalidate cache after DB commit: safer than writing a computed value, but still leaves stale cache if invalidation fails.
- Worst-case staleness is not automatically zero. It is bounded by TTL only if cached entries also have TTLs; otherwise stale data can persist until manual repair or overwrite.
- Replication lag still matters if reads refill the cache from SQL replicas after invalidation.

### Operational complexity
- Medium complexity.
- No separate CDC infrastructure is required, but every write path must correctly update or invalidate all affected cache keys.
- Hard parts:
  - identifying all derived keys affected by a write;
  - handling partial failures between SQL and Redis;
  - retrying cache updates safely;
  - preventing race conditions where an old reader repopulates stale data after invalidation;
  - keeping all code paths consistent, including admin jobs, batch imports, scripts, and migrations.

### Dominant failure modes
- Cache outage: writes may fail if the system treats cache update as required, or writes may proceed while cache updates are skipped, creating stale cache. If reads fall back to DB, DB load increases.
- DB outage: writes fail; cache may still serve old values. If the application writes cache before DB, it can expose uncommitted or false data.
- Replication lag: after cache deletion, a read may refill from a stale replica and cache old data again.
- Bug in write path: high blast radius. Missing one affected key or one write path can leave stale data for that key until TTL/manual repair.

### Right choice when
- The application has a small number of well-understood write paths.
- The set of affected cache keys per write is simple.
- You need better freshness than TTL-only but do not want CDC infrastructure.
- You can tolerate or mitigate partial failure windows using TTLs, retries, idempotent invalidation, and monitoring.

### Wrong choice when
- Many services, jobs, or direct DB writes can modify the data.
- Cache keys are numerous, derived, or hard to enumerate.
- You need strong correctness but cannot implement robust failure handling across DB and Redis.

## 3. Event-driven invalidation via CDC or pub/sub

In this strategy, writes go to SQL. A change stream, outbox, CDC process, or pub/sub event invalidates or updates affected Redis keys asynchronously.

### Correctness properties
- Readers can see stale values between DB commit and event processing.
- Normal staleness is approximately the CDC/pub-sub propagation and consumer processing lag.
- Under healthy operation, this can be milliseconds to seconds, but it is not zero.
- Under failure, staleness can last much longer:
  - if consumers are down, until they catch up;
  - if events are lost, until TTL, manual repair, reconciliation, or a future write;
  - if no TTL/backstop exists, stale data can be effectively unbounded.
- Correctness depends heavily on event delivery guarantees, replayability, ordering, idempotency, and key-mapping logic.

### Operational complexity
- Highest infrastructure complexity among the first three.
- Requires some combination of:
  - SQL CDC, transaction log tailing, Debezium, native DB replication feed, or outbox table;
  - message broker or stream such as Kafka, Redis Streams, SQS, Pub/Sub, etc.;
  - invalidation consumers;
  - dead-letter handling, replay, lag monitoring, and schema evolution handling.
- The benefit is decoupling writes from cache invalidation and capturing changes even if they come from multiple services or non-HTTP write paths.

### Dominant failure modes
- Cache outage: invalidation/update consumers fail to write to Redis or skip work. On recovery, they must retry or replay; otherwise stale entries remain.
- DB outage: writes stop, CDC stops producing new changes; existing cache values may continue to serve but become older.
- Replication/CDC lag: directly increases staleness. Backlogs can make stale reads last seconds, minutes, or longer.
- Bug in write path: if CDC observes actual DB commits, it can still invalidate even when a write path forgets to touch cache. However, if the bug writes incorrect data to DB, the cache will reflect or preserve that incorrectness. Bugs in event key mapping can leave affected keys stale.

### Right choice when
- Multiple services or tools write to the database.
- You need near-real-time invalidation but can tolerate asynchronous consistency.
- You already operate CDC or a durable event platform.
- You need a central, auditable invalidation pipeline.
- The product has grown beyond simple application-local write-through logic.

### Wrong choice when
- The team lacks operational maturity for CDC, stream processing, replays, and monitoring.
- The data requires immediate strong consistency.
- The system is small enough that TTL-only or simple write-through is sufficient.

## 4. Read-modify-write with version or ETag stored alongside the cached value

Here the cache stores something like `(value, version)`. SQL stores the authoritative version. Writes update the DB and bump the version. Reads compare versions and revalidate when mismatched.

There are two important variants:
1. Stronger variant: each read checks the authoritative DB version or a strongly consistent version source before trusting the cached value.
2. Weaker variant: the version is only stored in Redis or updated asynchronously, which does not fully prevent stale reads.

### Correctness properties
- If every read compares the cached version with the authoritative SQL version, stale values are detected before being returned. This can provide strong freshness for cached objects, but it adds a DB/version lookup to reads.
- If version checks are skipped, cached, asynchronous, or performed against a lagging replica, stale reads are still possible.
- Worst-case staleness:
  - with authoritative version check on every read: stale cached values should not be returned after the version bump is visible to the reader;
  - with replica-based version checks: staleness is bounded by replication lag plus any cache/version refresh behavior;
  - with no reliable authoritative check: stale data can persist until TTL or repair.
- This strategy is also useful for conditional writes, optimistic concurrency control, and HTTP ETag/If-Match semantics.

### Operational complexity
- Medium to high complexity.
- Requires:
  - version column or monotonically increasing revision in SQL;
  - cache format including value and version;
  - read logic that checks versions and refreshes on mismatch;
  - write logic that bumps versions correctly;
  - careful handling of concurrent updates.
- It can reduce stale reads but may reduce cache effectiveness because reads may still need a DB query for the version.

### Dominant failure modes
- Cache outage: application can fall back to DB; cache benefit disappears and DB load rises.
- DB outage: if reads require DB version checks, reads may fail even if Redis has a value. If serving cached values during DB outage is allowed, the system must explicitly accept potential staleness.
- Replication lag: checking versions against a lagging replica can falsely validate stale cached values.
- Bug in write path: if a write updates data but does not bump the version, stale cached values may be incorrectly treated as valid. This is a severe correctness bug.

### Right choice when
- You need stronger read freshness than TTL-only but can afford some DB involvement on reads.
- Objects have clear per-row or per-resource versions.
- You need optimistic concurrency control or HTTP ETag behavior anyway.
- Stale reads are problematic, but full distributed transactions are unnecessary or impractical.

### Wrong choice when
- Read volume is so high that even lightweight DB version checks are too expensive.
- Versions cannot be reliably bumped on every relevant change.
- Reads use lagging replicas but require strict freshness.
- The data is highly derived from many rows, making a single version hard to define.

## Comparison table

| Strategy | Staleness bound | Infra cost | Failure blast radius | Implementation effort |
|---|---:|---:|---:|---:|
| TTL-only | Normally up to TTL; longer if TTL missing/buggy; stale replica fills can cache old data for another TTL | Low | Usually limited to stale reads and DB load spike on cache outage; DB protected only while cache entries survive | Low |
| Write-through | Near-zero in happy path, but stale until TTL/retry/manual repair if DB commit succeeds and cache update/invalidation fails | Medium | Write-path bugs or Redis failures can affect all keys touched by that path; partial failure can create stale or false cache entries | Medium |
| Event-driven CDC/pub-sub | Normally event propagation lag; under consumer/broker/event-loss failure, until replay/catch-up/TTL/manual repair; unbounded without backstop | High | Pipeline outage can affect many keys/services; lag grows system-wide; durable replay reduces risk | High |
| Version/ETag with cached `(value, version)` | If checking authoritative version on each read: bounded by visibility of DB version bump; with replica/asynchronous checks: replication lag or TTL; if version not bumped: unbounded | Medium | DB/version-source outage can break reads or force stale-serving mode; write bug that misses version bump is severe | Medium to high |

## Recommendation for a greenfield product team with no strong consistency requirement

Start with TTL-only cache-aside, plus conservative engineering safeguards:
- choose short-to-moderate TTLs based on business tolerance for staleness;
- add TTL jitter to avoid thundering herds;
- protect the DB with request coalescing/single-flight for hot keys;
- monitor cache hit rate, Redis errors, DB load, and stale-data complaints;
- keep TTLs on all cache entries so stale data has a backstop;
- avoid caching authorization, payment, inventory, or other correctness-critical decisions unless separately validated.

If freshness requirements increase, the next step is usually explicit invalidation on the write path for simple domains, or CDC/event-driven invalidation once there are multiple writers or more complex data flows. Use version/ETag validation when stale reads are materially harmful and the team is willing to pay the extra read-path complexity or DB/version lookup cost.
