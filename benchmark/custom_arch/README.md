# Category C — Custom Architecture & Reasoning Tasks

Five hand-curated tasks that complement BigCodeBench-Hard (Category B) and
HumanEval+/SWE-bench (Category A). They represent the kind of coding work where
**functional tests do not apply but quality is still measurable**:

| # | ID | Kind |
|---|----|------|
| 1 | `custom-arch/auth-multitenant-design` | architecture-design |
| 2 | `custom-arch/migration-planning-zero-downtime` | migration-plan |
| 3 | `custom-arch/code-review-flaky-test` | code-review |
| 4 | `custom-arch/cache-invalidation-tradeoffs` | tradeoff-explanation |
| 5 | `custom-arch/production-debug-reasoning` | debug-reasoning |

## Why a separate category

Categories A and B can be scored by running the produced code against tests.
Category C cannot: there is no unit test for "is this migration plan sound" or
"does this debugging chain hold up". These tasks probe the dimensions of a
model's output that hybrid routing most plausibly affects — depth of reasoning,
completeness, practicality — and that are invisible to a pass/fail scorer.

The design goal when picking five tasks was **spread across the
interesting-for-hybrid-routing space**, not coverage of a domain. Each `kind`
appears exactly once.

## Scoring model

Every task ships with a 5-dimension rubric, each dimension scored out of 5:

- **correctness** — are the technical claims accurate?
- **completeness** — does the answer cover what the task asks?
- **style** — is it structured, readable, and appropriately terse?
- **reasoning_depth** — does it justify choices and surface trade-offs?
- **practicality** — could a real engineer act on this tomorrow?

Rubric descriptions are **tuned per task** — e.g. the correctness bar for a
debug walkthrough is different from the correctness bar for SQL DDL. This
matters because the LLM-judge in T3.3 consumes the per-task rubric verbatim.

The judge runs pairwise (cloud-only vs hybrid-architect output, blinded) and
awards each dimension independently. See T3.3 for the scorer implementation.

## Authoring notes

The tasks were written by hand, not generated by an LLM. That is deliberate:
if the benchmark were generated by the same class of model being evaluated, it
would bias toward prompts that model happens to be good at. The credibility of
this category rests on the tasks being independent of any model's output
distribution.

`expected_topics` on each task is a rough list of what a good answer hits. It
is **not** the scoring key — the rubric is — but it gives the judge (and a
human auditor, per T4.4) a sanity anchor.
